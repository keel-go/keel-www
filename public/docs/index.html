Keel Documentation

A zero-dependency Go web framework where every feature is a self-describing module. Explicit, agent-friendly, compile-time safe.

Existing Go frameworks either do too little (just routing) or too much (magic, reflection, tons of dependencies). Keel sits in between: full-stack features with Go's explicitness.

The core idea: every module fully declares itself. No service providers. No reflection. No scattered config files. No init() side effects. A module is a Go struct implementing an interface. The framework is the sum of its modules.

This has three practical consequences:

  • Human-readable. Open one module.go and see every route, model, migration, event, and dependency for that feature.
  • Agent-friendly. A coding agent reads one file per module and understands the full scope. No hunting through config.
  • Compile-time safe. If it compiles, the wiring is correct. No runtime registration surprises.

Installation

Install Keel with go get:

bash 
go get github.com/keel-go/keel

Requires Go 1.21+ for log/slog support. Keel has zero external dependencies — only Go stdlib and golang.org/x/ packages.

Quickstart

Create a minimal app with one module:

go 
package main

import (
    "github.com/keel-go/keel"
)

// GreetModule is a minimal module that serves a greeting.
type GreetModule struct{}

func (m GreetModule) Name() string                   { return "greet" }
func (m GreetModule) DependsOn() []string            { return nil }
func (m GreetModule) Models() []keel.Model           { return nil }
func (m GreetModule) Migrations() []keel.Migration   { return nil }
func (m GreetModule) Events() []keel.EventBinding    { return nil }

func (m GreetModule) Routes() []keel.Route {
    return []keel.Route{
        keel.Get("/hello/{name}", m.Hello),
    }
}

func (m GreetModule) Hello(ctx *keel.Context) error {
    name := ctx.Param("name")
    return ctx.JSON(200, map[string]string{"message": "Hello, " + name + "!"})
}

func main() {
    app := keel.New(GreetModule{})
    app.Use(keel.Logger())
    app.Use(keel.Recovery())
    app.RunCLI() // go run . serve
}

Run it:

bash 
go run . serve
# => Keel listening on :8080

curl http://localhost:8080/hello/world
# => {"message":"Hello, world!"}

Project Structure

A Keel application follows a modular directory layout. Each feature gets its own directory under modules/:

myapp/ ├── main.go # App bootstrap — lists all modules ├── .env # Environment variables ├── modules/ │ ├── user/ │ │ ├── module.go # Module struct — routes, models, events, deps │ │ ├── handlers.go # HTTP handler methods │ │ ├── models.go # User struct, CreateUserInput, etc. │ │ ├── migrations.go # Schema definitions │ │ └── events.go # UserCreatedEvent, UserDeletedEvent │ ├── blog/ │ │ ├── module.go │ │ ├── handlers.go │ │ ├── models.go │ │ ├── migrations.go │ │ └── events.go │ └── notification/ │ ├── module.go │ ├── handlers.go │ └── listeners.go # Reacts to events from other modules └── public/ └── assets/

Framework Package Structure

The Keel framework itself is organized as a single Go package:

keel/ # Framework package ├── app.go # App struct, New(), Serve(), RunCLI() ├── module.go # Module interface + optional interfaces ├── router.go # Radix-tree HTTP router ├── context.go # Request/response context ├── middleware.go # Middleware types + built-in middleware ├── db.go # Database connection manager ├── query.go # Query builder ├── model.go # Model definition + relationships ├── migrate.go # Migration runner + schema builder ├── validate.go # Validation engine ├── event.go # Event bus (sync + async dispatch) ├── session.go # Session management ├── jwt.go # JWT creation/validation (HMAC-SHA256) ├── config.go # Env var loader + typed config ├── cli.go # CLI command runner ├── log.go # Structured logging (slog wrapper) ├── storage.go # File storage abstraction ├── ws.go # WebSocket support ├── sse.go # Server-Sent Events ├── testing.go # Test app builder + assertions └── errors.go # Error types (HTTPError, ValidationError)

Modules

Modules are the foundational concept in Keel. Every feature in your application is a module — a Go struct that implements the Module interface. A module fully declares its routes, models, migrations, events, and dependencies in one place.

Here's a complete blog module:

go 
package blog

import "github.com/keel-go/keel"

type Module struct{}

func (m Module) Name() string { return "blog" }

func (m Module) Routes() []keel.Route {
    return []keel.Route{
        keel.Get("/posts", m.ListPosts),
        keel.Get("/posts/{id}", m.ShowPost),
        keel.Post("/posts", m.CreatePost).Middleware(keel.Auth),
        keel.Delete("/posts/{id}", m.DeletePost).Middleware(keel.Auth),
    }
}

func (m Module) Models() []keel.Model {
    return []keel.Model{
        keel.DefineModel("posts", Post{},
            keel.BelongsTo("author", user.User{}),
            keel.HasMany("comments", Comment{}),
        ),
    }
}

func (m Module) Migrations() []keel.Migration {
    return []keel.Migration{
        {Version: 1, Up: m.createPostsTable, Down: m.dropPostsTable},
    }
}

func (m Module) Events() []keel.EventBinding {
    return []keel.EventBinding{
        {Emits: PostCreatedEvent{}},
        {Emits: PostDeletedEvent{}},
        {Listens: user.UserDeletedEvent{}, Handler: m.cleanupUserPosts},
    }
}

func (m Module) DependsOn() []string {
    return []string{"user"}
}

One module = one module.go = complete declaration. A developer or AI agent can open a single file to understand the full scope of any feature.

Module Interface

This is the entire contract. Every feature in a Keel app implements this interface:

go 
type Module interface {
    Name() string
    Routes() []Route
    Models() []Model
    Migrations() []Migration
    Events() []EventBinding
    DependsOn() []string
}
Method Returns Purpose
Name() string Unique identifier for the module
Routes() []Route HTTP routes this module handles
Models() []Model Database models with relationships
Migrations() []Migration Schema migrations (up/down)
Events() []EventBinding Events emitted and listened to
DependsOn() []string Names of modules this one requires

Return nil for any method that doesn't apply to your module. For example, a module that only provides routes with no database models:

go 
func (m HealthModule) Models() []keel.Model         { return nil }
func (m HealthModule) Migrations() []keel.Migration { return nil }
func (m HealthModule) Events() []keel.EventBinding  { return nil }
func (m HealthModule) DependsOn() []string          { return nil }

Optional Interfaces

Beyond the core Module interface, Keel provides optional interfaces you can implement to extend module behavior. Only implement what you need:

go 
type HasMiddleware interface{ Middleware() []Middleware }
type HasCommands   interface{ Commands() []Command }
type HasSchedule   interface{ Schedule() []ScheduledTask }
type HasConfig     interface{ Config() any }
type HasSeeder     interface{ Seed(db *DB) error }
Interface Method Purpose
HasMiddleware Middleware() []Middleware Module-scoped middleware applied to all routes
HasCommands Commands() []Command Custom CLI commands for this module
HasSchedule Schedule() []ScheduledTask Recurring tasks (cron-style scheduling)
HasConfig Config() any Typed configuration struct for the module
HasSeeder Seed(db *DB) error Database seeder for test/development data

Example: Module with Middleware and Commands

go 
type AdminModule struct{}

// Core Module interface methods...

// HasMiddleware — all admin routes require auth
func (m AdminModule) Middleware() []keel.Middleware {
    return []keel.Middleware{keel.Auth, keel.RequireRole("admin")}
}

// HasCommands — custom CLI commands
func (m AdminModule) Commands() []keel.Command {
    return []keel.Command{
        {Name: "admin:reset", Desc: "Reset admin password", Run: m.resetAdmin},
    }
}

Lifecycle Hooks

Modules can hook into the application lifecycle by implementing these interfaces:

go 
type OnInit     interface{ Init(app *App) error }
type OnReady    interface{ Ready(app *App) error }
type OnShutdown interface{ Shutdown(ctx context.Context) error }
Hook When Use Case
OnInit During app.New(), before the server starts Set up connections, validate config, register services
OnReady After all modules are initialized, before serving Start background workers, warm caches
OnShutdown On graceful shutdown (SIGTERM/SIGINT) Close connections, flush buffers, cleanup
go 
func (m CacheModule) Init(app *keel.App) error {
    // Connect to Redis, set up cache pools, etc.
    m.pool = initRedis(app.Config("REDIS_URL"))
    return nil
}

func (m CacheModule) Shutdown(ctx context.Context) error {
    return m.pool.Close()
}

Routing

Keel uses a radix-tree router for fast route matching. Routes are defined in a module's Routes() method using convenience functions:

go 
func (m Module) Routes() []keel.Route {
    return []keel.Route{
        keel.Get("/posts", m.ListPosts),
        keel.Get("/posts/{id}", m.ShowPost),
        keel.Post("/posts", m.CreatePost),
        keel.Put("/posts/{id}", m.UpdatePost),
        keel.Patch("/posts/{id}", m.PatchPost),
        keel.Delete("/posts/{id}", m.DeletePost),
    }
}
Function HTTP Method Example
keel.Get(path, handler) GET Read resources, list items
keel.Post(path, handler) POST Create new resources
keel.Put(path, handler) PUT Replace entire resource
keel.Patch(path, handler) PATCH Partial resource update
keel.Delete(path, handler) DELETE Remove resource

Route-Level Middleware

Chain middleware on individual routes using the .Middleware() method:

go 
keel.Post("/posts", m.CreatePost).Middleware(keel.Auth)
keel.Delete("/posts/{id}", m.DeletePost).Middleware(keel.Auth, keel.RequireRole("admin"))

Route Parameters

Define dynamic segments in your route paths using curly braces. Access them via ctx.Param():

go 
// Route: keel.Get("/posts/{id}", m.ShowPost)

func (m Module) ShowPost(ctx *keel.Context) error {
    id := ctx.Param("id")   // string value from URL
    // ...
}

// Nested parameters: /users/{userId}/posts/{postId}
func (m Module) UserPost(ctx *keel.Context) error {
    userId := ctx.Param("userId")
    postId := ctx.Param("postId")
    // ...
}

Route Groups

While Keel doesn't have explicit route groups (each module is itself a group), you can organize routes logically within a module and apply middleware at the module level via the HasMiddleware interface:

go 
// All routes in this module get auth middleware applied
func (m AdminModule) Middleware() []keel.Middleware {
    return []keel.Middleware{keel.Auth, keel.RequireRole("admin")}
}

func (m AdminModule) Routes() []keel.Route {
    return []keel.Route{
        keel.Get("/admin/users", m.ListUsers),
        keel.Get("/admin/stats", m.Stats),
        keel.Post("/admin/settings", m.UpdateSettings),
    }
}

Context

Every handler receives a *keel.Context that wraps the HTTP request and response, providing helpers for common operations. Handlers return an error — the framework handles error responses automatically.

go 
func (m Module) CreatePost(ctx *keel.Context) error {
    var input CreatePostInput
    if err := ctx.BindJSON(&input); err != nil {
        return err
    }
    if err := ctx.Validate(input); err != nil {
        return err // returns 422 with structured errors
    }
    post, err := ctx.DB.Create("posts", map[string]any{
        "title":   input.Title,
        "body":    input.Body,
        "user_id": ctx.UserID(),
    })
    if err != nil {
        return err
    }
    ctx.Emit(PostCreatedEvent{Post: post})
    return ctx.JSON(201, post)
}

Request Helpers

The context provides methods for reading data from the incoming request:

Method Returns Description
ctx.Param(name) string URL path parameter
ctx.Query(name) string Query string parameter
ctx.QueryInt(name, default) int Query parameter parsed as integer
ctx.Header(name) string Request header value
ctx.BindJSON(&v) error Decode JSON body into struct
ctx.FormValue(name) string Form field value
ctx.FormFile(name) *UploadedFile, error Uploaded file from multipart form
ctx.UserID() string Authenticated user ID (from JWT/session)
ctx.Validate(v) error Validate a struct using defined rules
go 
func (m Module) ListPosts(ctx *keel.Context) error {
    page := ctx.QueryInt("page", 1)
    search := ctx.Query("q")
    userId := ctx.Param("userId")

    posts, err := ctx.DB.Select("posts").
        Where("user_id = ?", userId).
        With("author").
        Paginate(page, 20)
    if err != nil {
        return err
    }
    return ctx.JSON(200, posts)
}

Response Helpers

The context provides methods for sending responses:

Method Description
ctx.JSON(status, data) Send JSON response with status code
ctx.String(status, text) Send plain text response
ctx.HTML(status, html) Send HTML response
ctx.NoContent(status) Send empty response (e.g., 204)
ctx.Redirect(status, url) Redirect to another URL
ctx.SetHeader(name, value) Set a response header
ctx.SetCookie(cookie) Set a response cookie

Error Responses

Returning an error from a handler automatically generates the appropriate HTTP response:

go 
// Return a generic error → 500 Internal Server Error
return fmt.Errorf("something went wrong")

// Return an HTTPError for specific status codes
return keel.NewHTTPError(404, "Post not found")
return keel.NewHTTPError(403, "Forbidden")

// Validation errors → 422 with structured field errors
return ctx.Validate(input) // auto-returns ValidationError

Database

Keel includes a built-in database connection manager. Access it through the context in handlers via ctx.DB:

go 
// Database connection is configured via environment variables
// DB_DRIVER=postgres
// DB_DSN=postgres://user:pass@localhost/myapp?sslmode=disable

func (m Module) ListPosts(ctx *keel.Context) error {
    posts, err := ctx.DB.Select("posts").All()
    if err != nil {
        return err
    }
    return ctx.JSON(200, posts)
}

Query Builder

The built-in query builder provides a fluent API for constructing database queries without raw SQL:

go 
// SELECT — fetch records
posts, err := ctx.DB.Select("posts").
    Where("published = ?", true).
    OrderBy("created_at DESC").
    Limit(10).
    All()

// SELECT with eager loading
posts, err := ctx.DB.Select("posts").
    With("author").
    Paginate(page, 20)

// SELECT single record
post, err := ctx.DB.Select("posts").
    Where("id = ?", id).
    First()

// CREATE — insert a record
post, err := ctx.DB.Create("posts", map[string]any{
    "title":   "My Post",
    "body":    "Content here",
    "user_id": userId,
})

// UPDATE — modify records
err := ctx.DB.Update("posts", id, map[string]any{
    "title": "Updated Title",
})

// DELETE — remove records
err := ctx.DB.Delete("posts", id)

Pagination

The Paginate(page, perPage) method returns a paginated result set:

go 
page := ctx.QueryInt("page", 1)
results, err := ctx.DB.Select("posts").Paginate(page, 20)
// results includes: data, total, page, per_page, last_page

Models

Models are plain Go structs. Keel maps struct fields to database columns using snake_case convention — no struct tags required for framework behavior:

go 
type Post struct {
    ID        int64      // → id
    Title     string     // → title
    Body      string     // → body
    UserID    int64      // → user_id
    Published bool       // → published
    CreatedAt time.Time  // → created_at
    UpdatedAt time.Time  // → updated_at
}

type CreatePostInput struct {
    Title string `json:"title" validate:"required,max=200"`
    Body  string `json:"body"  validate:"required"`
}

Register models in your module's Models() method:

go 
func (m Module) Models() []keel.Model {
    return []keel.Model{
        keel.DefineModel("posts", Post{}),
        keel.DefineModel("comments", Comment{}),
    }
}

Relationships

Define model relationships inline when registering models:

go 
keel.DefineModel("posts", Post{},
    keel.BelongsTo("author", user.User{}),    // post belongs to a user
    keel.HasMany("comments", Comment{}),      // post has many comments
    keel.HasOne("metadata", PostMeta{}),     // post has one metadata
)
Function Description Foreign Key
BelongsTo(name, model) This model has a foreign key to the related model user_id on posts table
HasMany(name, model) Related model has many references to this one post_id on comments table
HasOne(name, model) Related model has exactly one reference to this one post_id on metadata table

Eager-load relationships using .With():

go 
posts, err := ctx.DB.Select("posts").With("author").With("comments").All()

Migrations

Define versioned, reversible migrations in your module. Each migration has an Up function (to apply) and a Down function (to rollback):

go 
func (m Module) Migrations() []keel.Migration {
    return []keel.Migration{
        {Version: 1, Up: m.createPostsTable, Down: m.dropPostsTable},
        {Version: 2, Up: m.addPublishedColumn, Down: m.removePublishedColumn},
    }
}

func (m Module) createPostsTable(schema *keel.Schema) {
    schema.Create("posts", func(t *keel.Table) {
        t.ID()
        t.String("title", 200)
        t.Text("body")
        t.ForeignKey("user_id", "users")
        t.Bool("published").Default(false)
        t.Timestamps()
    })
}

func (m Module) dropPostsTable(schema *keel.Schema) {
    schema.Drop("posts")
}

Running Migrations

bash 
go run . migrate up       # Run all pending migrations
go run . migrate down     # Rollback the last migration
go run . migrate status   # Show migration state per module

Authentication

Keel provides built-in JWT (HMAC-SHA256) and session-based authentication. No external dependencies required.

JWT Authentication

go 
// Generate a JWT token
token, err := keel.NewJWT(keel.JWTClaims{
    UserID: user.ID,
    Role:   user.Role,
}, app.Config("JWT_SECRET"))

// Return it to the client
return ctx.JSON(200, map[string]string{"token": token})

Protecting Routes

go 
// Use keel.Auth middleware to protect routes
keel.Post("/posts", m.CreatePost).Middleware(keel.Auth)

// Access the authenticated user in handlers
func (m Module) CreatePost(ctx *keel.Context) error {
    userId := ctx.UserID()   // extracted from JWT
    // ...
}

// Role-based access control
keel.Delete("/admin/users/{id}", m.DeleteUser).
    Middleware(keel.Auth, keel.RequireRole("admin"))

Session Management

go 
// Store data in session
ctx.Session.Set("user_id", user.ID)
ctx.Session.Save()

// Read from session
userId := ctx.Session.Get("user_id")

// Destroy session (logout)
ctx.Session.Destroy()

Middleware

Middleware wraps handlers to add cross-cutting behavior. Keel supports middleware at three levels: global, module, and route.

Built-in Middleware

Middleware Description
keel.Logger() Structured request logging (method, path, status, duration)
keel.Recovery() Recover from panics and return 500 error
keel.CORS(config) Cross-Origin Resource Sharing headers
keel.Auth JWT authentication (reads Authorization header)
keel.RequireRole(role) Role-based access control (use after Auth)

Global Middleware

Apply middleware to all routes in the application:

go 
app := keel.New(modules...)
app.Use(keel.Logger())
app.Use(keel.Recovery())
app.Use(keel.CORS(keel.CORSConfig{Origin: "*"}))

Module-Level Middleware

Apply to all routes within a module by implementing HasMiddleware:

go 
func (m AdminModule) Middleware() []keel.Middleware {
    return []keel.Middleware{keel.Auth}
}

Custom Middleware

go 
func RateLimit(requests int, window time.Duration) keel.Middleware {
    return func(next keel.Handler) keel.Handler {
        return func(ctx *keel.Context) error {
            // Check rate limit...
            if overLimit {
                return keel.NewHTTPError(429, "Too many requests")
            }
            return next(ctx)
        }
    }
}

Event Bus

Keel includes a built-in event bus for decoupling modules. Events support both synchronous and asynchronous dispatch.

Defining Events

go 
// Define event types as plain structs
type PostCreatedEvent struct {
    Post Post
}

type PostDeletedEvent struct {
    PostID int64
}

Declaring Events in a Module

Declare which events a module emits and listens to in the Events() method:

go 
func (m Module) Events() []keel.EventBinding {
    return []keel.EventBinding{
        // This module emits these events
        {Emits: PostCreatedEvent{}},
        {Emits: PostDeletedEvent{}},

        // This module listens to events from other modules
        {Listens: user.UserDeletedEvent{}, Handler: m.cleanupUserPosts},
    }
}

Emitting Events

go 
// Emit an event from a handler
ctx.Emit(PostCreatedEvent{Post: post})

Handling Events

go 
func (m NotificationModule) onPostCreated(event any) error {
    e := event.(PostCreatedEvent)
    // Send notification, update feeds, etc.
    return nil
}

Events are a great way to keep modules decoupled. The blog module doesn't need to know about notifications — it just emits PostCreatedEvent, and any module can listen.

Validation

Keel provides built-in validation with explicit rules. Validate incoming data using ctx.Validate():

go 
type CreatePostInput struct {
    Title string `json:"title" validate:"required,max=200"`
    Body  string `json:"body"  validate:"required"`
    Tags  []string `json:"tags" validate:"max=5"`
}

func (m Module) CreatePost(ctx *keel.Context) error {
    var input CreatePostInput
    if err := ctx.BindJSON(&input); err != nil {
        return err
    }
    if err := ctx.Validate(input); err != nil {
        return err // returns 422 with structured field errors
    }
    // input is valid, proceed...
}

Validation Error Response

When validation fails, a 422 response is returned with structured errors:

json 
{
  "error": "Validation failed",
  "fields": {
    "title": ["title is required"],
    "body": ["body is required"]
  }
}

Available Validation Rules

Rule Description Example
required Field must be present and non-empty validate:"required"
min=N Minimum length (string) or value (number) validate:"min=3"
max=N Maximum length (string) or value (number) validate:"max=200"
email Must be a valid email address validate:"email"
url Must be a valid URL validate:"url"
oneof=a b c Must be one of the listed values validate:"oneof=draft published"

WebSocket

Keel provides built-in WebSocket support for real-time bidirectional communication:

go 
func (m Module) Routes() []keel.Route {
    return []keel.Route{
        keel.Get("/ws", m.HandleWebSocket),
    }
}

func (m Module) HandleWebSocket(ctx *keel.Context) error {
    conn, err := ctx.UpgradeWebSocket()
    if err != nil {
        return err
    }
    defer conn.Close()

    for {
        msg, err := conn.ReadMessage()
        if err != nil {
            break
        }
        // Echo message back
        conn.WriteMessage(msg)
    }
    return nil
}

Server-Sent Events

SSE provides a simple way to push real-time updates from server to client over HTTP:

go 
func (m Module) StreamUpdates(ctx *keel.Context) error {
    stream, err := ctx.SSE()
    if err != nil {
        return err
    }
    defer stream.Close()

    for {
        select {
        case <-ctx.Done():
            return nil
        case update := <-m.updates:
            stream.Send("update", update)
        }
    }
}

Client-Side

javascript 
const source = new EventSource('/stream');
source.addEventListener('update', (e) => {
  const data = JSON.parse(e.data);
  console.log('Received:', data);
});

File Storage

Keel provides a file storage abstraction for handling uploads and file management:

go 
func (m Module) UploadAvatar(ctx *keel.Context) error {
    file, err := ctx.FormFile("avatar")
    if err != nil {
        return err
    }

    path, err := ctx.Storage.Put("avatars", file)
    if err != nil {
        return err
    }

    return ctx.JSON(200, map[string]string{"path": path})
}

// Read a file
data, err := ctx.Storage.Get("avatars/user-123.jpg")

// Delete a file
err := ctx.Storage.Delete("avatars/user-123.jpg")

// Check if file exists
exists := ctx.Storage.Exists("avatars/user-123.jpg")

Storage backends are configured via environment variables. Keel supports local filesystem storage out of the box. Configure STORAGE_DRIVER and STORAGE_PATH in your .env file.

Logging

Keel wraps Go's log/slog (available since Go 1.21) to provide structured logging:

go 
// In handlers, use the context logger
ctx.Log.Info("post created", "post_id", post.ID, "user_id", userId)
ctx.Log.Error("failed to send email", "err", err)
ctx.Log.Warn("rate limit approaching", "count", count)
ctx.Log.Debug("processing request", "path", ctx.Path())

The Logger() middleware automatically logs every request:

text 
2024-01-15T10:30:00Z INF request method=GET path=/posts status=200 duration=2.3ms
2024-01-15T10:30:01Z INF request method=POST path=/posts status=201 duration=15.7ms
2024-01-15T10:30:02Z ERR request method=GET path=/posts/999 status=404 duration=1.1ms

Accessing Config

go 
// Simple string config
port := app.Config("APP_PORT")

// With default value

Module-Level Config

Modules can declare typed configuration by implementing the HasConfig interface:

go 
type EmailConfig struct {
    SMTPHost string `env:"SMTP_HOST"`
    SMTPPort int    `env:"SMTP_PORT" default:"587"`
    From     string `env:"EMAIL_FROM"`
}

func (m EmailModule) Config() any {
    return &EmailConfig{}
}

Agent Workflow

Keel is designed to be AI-agent-friendly. An AI coding agent working with Keel follows this loop:

  1. Read main.go — see all modules, understand app scope
  2. Read target module.go — see routes, models, events, deps for that feature
  3. Run go run . describe — get machine-readable JSON of entire app (optional, for complex tasks)
  4. Generate code — new modules follow the identical Module interface pattern every time
  5. Compile — go build ./... catches all errors instantly (no runtime surprises)
  6. Test — go test ./... with keel.TestApp() for integration tests

An agent never needs to search for where something is defined. One module = one module.go = complete declaration. No hunting through scattered config files or service providers.

Design Constraints

Keel enforces these deliberate constraints to keep codebases predictable and maintainable:

Constraint Rationale
Zero external dependencies Only Go stdlib and golang.org/x/ packages. No transitive dependency risk.
No reflection All wiring is explicit Go code. No runtime type inspection surprises.
No struct tags for framework behavior Models use convention (snake_case field mapping). Tags are only for JSON/validation.
No global state Everything hangs off the App struct. No package-level vars or singletons.
No init() functions Module initialization is explicit via lifecycle hooks (OnInit, OnReady).
Compile-time safety If it compiles, the wiring is correct. No runtime registration surprises.

Complete Handler Example

go 
func (m Module) ListPosts(ctx *keel.Context) error {
    posts, err := ctx.DB.Select("posts").
        With("author").
        Paginate(ctx.QueryInt("page", 1), 20)
    if err != nil {
        return err
    }
    return ctx.JSON(200, posts)
}

func (m Module) CreatePost(ctx *keel.Context) error {
    var input CreatePostInput
    if err := ctx.BindJSON(&input); err != nil {
        return err
    }
    if err := ctx.Validate(input); err != nil {
        return err // returns 422 with structured errors
    }

    post, err := ctx.DB.Create("posts", map[string]any{
        "title":   input.Title,
        "body":    input.Body,
        "user_id": ctx.UserID(),
    })
    if err != nil {
        return err
    }

    ctx.Emit(PostCreatedEvent{Post: post})
    return ctx.JSON(201, post)
}
